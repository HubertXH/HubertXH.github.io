Redis AOF持久话

1. AOF持久化概述

AOF(Append Only File)通过记录写命令来实现数据持久化，具有更高的数据安全性。

1.1 工作原理示例

AOF通过将所有写命令按顺序追加到一个文件中来记录数据库状态。当Redis重启时，会读取AOF文件并重新执行这些命令，从而恢复数据库状态。这种方式保证了数据的一致性，即使Redis异常退出，也能最大程度地恢复数据。
例：

  • AOF方式：记录HSET、SADD这三条命令

2. AOF持久化实现机制

2.1 基本流程

  1. 命令追加：当AOF持久化功能开启时，服务器执行完写命令后，会将该命令以协议格式追加到aof_buf缓冲区末尾
  2. 缓冲区刷新：服务器在每个事件循环结束前，会调用flushAppendOnlyFile函数，根据配置决定是否将缓冲区内容写入AOF文件

2.2 写入策略

Redis提供了三种AOF缓冲区写入策略，通过appendfsync选项配置：

策略 描述 优缺点
always 每个事件循环都同步写入AOF文件 最安全，但最慢
everysec 每秒同步一次 性能和安全性的平衡点，最多丢失1秒的数据
no 由操作系统决定同步时机 最快，但可能丢失上次同步之后的所有数据
  • 数据安全性优先： 选择always或everysec，保证数据不丢失。
  • 性能优先： 如果对数据丢失容忍度较高，可以选择no或everysec。
  • 混合使用： 可以同时开启RDB和AOF，RDB用于快速全量恢复，AOF用于保证数据不丢失。

3. AOF重写机制

AOF文件随着写入操作的增多，会不断增长。为了避免AOF文件过大影响性能，Redis引入了AOF重写机制。AOF重写并不是简单地删除旧的AOF文件，而是创建一个新的AOF文件，这个新的AOF文件只包含重建当前数据库所必需的命令，从而大大缩小了文件尺寸。

3.1 重写目的

  • 减小AOF文件体积
  • 提高数据恢复效率
  • 降低磁盘占用

3.2 重写实现原理

  • 不读取现有AOF文件
  • 通过读取当前数据库状态来实现
  • 用一条命令记录键值对，替代之前的多条命令

触发方式

  • 手动触发： 执行BGREWRITEAOF命令。
  • 自动触发： 当AOF文件大小超过配置的阈值（auto-aof-rewrite-percentage和auto-aof-rewrite-min-size）时，Redis会自动触发重写。

3.3 后台重写机制

重写流程

  1. 创建子进程：
    Redis主进程fork出一个子进程。子进程继承了父进程的几乎所有内容，包括内存中的数据库数据。
  2. 子进程重写：
    子进程开始遍历内存中的数据库，将数据库中的键值对转化为相应的写命令，并写入到一个新的临时AOF文件中。
    由于子进程只关心当前数据库的状态，因此新的AOF文件会比旧的AOF文件更精简。
  3. 替换旧文件：
    子进程完成重写后，向父进程发送信号。
    父进程接收到信号后，原子地用新的AOF文件替换旧的AOF文件。

优点

  1. 并发执行

    • 主进程继续处理客户端请求
    • 子进程专注于AOF重写

  2. 数据安全

    • 子进程拥有主进程数据副本
    • 避免使用锁机制

  3. 数据一致性保证

    • 设置AOF重写缓冲区
    • 写命令同时发送至AOF缓冲区和重写缓冲区
    • 新的AOF文件会先写入到一个临时文件中。只有在重写完成后，才会原子地将临时文件替换为原来的AOF文件

优点：整个过程仅在最后替换文件时短暂阻塞主进程，最大限度地降低了对性能的影响。

AOF的优势与劣势

  • 减少AOF文件大小：通过重写，可以大大缩小AOF文件的大小，提高磁盘利用率。
  • 提高性能：较小的AOF文件读取速度更快，从而提高Redis的性能。
  • 提高数据安全性：AOF重写可以减少冗余数据，提高数据恢复的效率。
