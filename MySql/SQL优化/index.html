使用explain查看SQL语句执行情况

示例：

EXPLAIN select * from product where category_id=51999 order by barcode desc limit 10;


说明：这个是一个简单的查询，数据都是从product表中查询获取的，使用的索引是category_idx，索引长度为4字节，
使用一个常量值和索引进行比较，总过扫描了211696行数据，通过条件过滤后所有的行都符合。使用了文件排序，因为barcode字段上没有索引。

字段说明：

字段 Meaning
id 语句的编号

Extra 附加信息
select type
类型 说明
SIMPLE 简单的查询
PRIMARY 表示查询中的外层查询（最外层查询）。当查询中包含子查询时，最外层的查询会被标记为 PRIMARY
UNION 表示查询使用了 UNION 操作符。当查询中包含多个 SELECT 语句并合并它们的结果时，MySQL 会把每个 SELECT 语句的 select_type 标记为 UNION
DEPENDENT UNION UNION 查询的某一部分依赖于外层查询的结果。也就是说，这个 UNION 子查询的执行依赖于外部查询的某个列值或条件
UNION RESULT 表示 UNION 查询的最终合并结果。这是执行 UNION 操作后返回的结果集部分，而不是每个子查询的部分。
SUBQUERY 表示查询中的子查询.在 SELECT、WHERE 或 FROM 子句中嵌套的查询部分通常会被标记为 SUBQUERY。
DEPENDENT SUBQUERY 表示依赖于外层查询的子查询。即该子查询的执行结果依赖于外部查询中的列值。
DERIVED 表示派生表（或临时表），通常是在 FROM 子句中的子查询。MySQL 会首先执行子查询并将其结果作为临时表（派生表）处理，再在外层查询中使用这个临时表。
DEPENDENT DERIVED 在执行该派生表的查询时，需要外层查询的某些数据作为输入来执行
MATERIALIZED 视图。MySQL 会将一个子查询的结果物化为临时表，在查询执行过程中使用这个临时表。
type

  • system 表中的数据只有一行，例如使用主键查询。system是const的一种特殊类型。

  • const 表示从查询一开始就知道最多能匹配到一条数据。性能是很棒的。

  • eq_ref MySQL 使用了唯一索引或主键，能够快速找到每一行的匹配项。

  • ref 索引查找，即 MySQL 使用索引查找符合条件的记录，但该索引可能有重复的值（即索引列不是唯一的）。他是查找和扫描的混合体，当只有访问非唯一索引或者唯一索引的前缀是会出现，。

  • ref_or_null 是ref的一个变形体，他表是Mysql必须在第一次的查询结果中进行二次查询来找到值为NULL的数据。

  • range 范围扫描，即 MySQL 使用索引查找符合特定范围条件（如 BETWEEN、<、> 等）的记录。它会扫描索引的一部分，而不是全表或整个索引。

  • index 索引扫描-即 MySQL 会扫描表的索引，而不是扫描表本身的数据。通常比通过索引查找的性能差一些。index 类型意味着 MySQL 扫描了索引树，但没有访问表中的数据行（也就是说，它访问的是索引本身）

  • all 全表扫描

    高效类型：const、eq_ref 和 system。
    中等效率类型：ref、range 和 index。
    低效类型：ALL（全表扫描）。

filtered

在查询执行过程中，通过某个步骤（如扫描、索引查找等）后，实际返回的符合条件的行数与表的数据量的百分比。
该值是一个估算值，通常是一个介于 0 到 100 之间的浮动值，表示经过该步骤后有多少比例的行被过滤掉。它计算了查询中条件（如 WHERE 子句）应用到查询结果集中的每一行时，有多少行会被保留下来。
值在0-100之间，接近 100：表示大多数行都符合条件，过滤较少。例如，当查询使用了非常有效的索引并且查询条件比较宽松时，filtered 值可能会接近 100。较低值（如 20 或更低）：表示很多行被过滤掉，只有少部分行符合条件。接近 0：表示几乎所有的行都被过滤掉了，符合条件的行非常少。通常，这意味着查询条件非常严格，或者表中的数据并不满足查询条件。

Extra
  • Using index 查询使用了覆盖索引（Covering Index）。这意味着查询只从索引中获取所有需要的列，而没有访问表中的数据行。
  • Using where 查询在检索数据时使用了 WHERE 条件进行过滤，单独出现时；一般代表表上出现全表扫描过滤
  • Using temporary MySQL 创建了一个临时表来存储中间结果。这通常发生在执行诸如 GROUP BY、ORDER BY 或连接操作时，无法通过索引完成这些操作时，
  • Using filesort MySQL 使用文件排序来完成 ORDER BY 操作，而不是通过索引进行排序。
  • Using join buffer 查询在执行连接操作时使用了连接缓冲区（Join Buffer）。这通常发生在执行没有使用索引的连接时。
  • Using index condition 通过索引初步过滤；回表再过滤其它条件；
  • Using index & Using where 使用索引返回数据；同时通过索引过滤。
  • Using all 查询扫描了所有的行（全表扫描）

常用的优化方法：

查询Select
  1. order by 使用的字段和Where条件使用相同的索引
  2. order by 字段的顺序和索引的顺序相同
  3. order by 字段同为升序或同为降序
  4. 避免使用select * 来进行查询，避免因字段变更发生的异常，只查询需要的字段 如果可以的话可以使用覆盖索引
  5. 对于子查询可以使用性能更好的join来替换
  6. 对于分页查询limit在业务允许的情况下使用主键范围来进行查询，或进行SQL改写
  7. 对于and条件子句多个条件中有一个拥有过滤性较高的索引即可，对于比较常用的and条件可以创建复合索引。
  8. 对于or条件子句前后条件都必须有索引。
  9. 对于join语句多表关联的字段需要有索引，而且应以小数量的表进行驱动来查询
写入

批量写入尽量使用一条SQL语句。当然 load data infile的性能最好

其他优化建议
  • where条件子句中字段值的类型需要和表中字段保持一致，否则会引起隐式转换导致索引失效
  • 索引模糊查询需要遵循最左匹配原则，否则索引失效
  • 禁止在索引字段列使用函数表达式或计算，否则索引失效
  • 联合索引使用的时候需要保证索引顺序，即索引中第一个字段需要在查询条件中，否则不会使用索引
  • where in 条件里面的值尽量要少
